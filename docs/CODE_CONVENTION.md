# 코드 컨벤션 (Python/FastAPI)

## 1. 기본 원칙

- 읽기 쉬운 코드가 짧은 코드보다 우선이다.
- 한 함수/메서드는 한 가지 책임만 가진다.
- 암묵적 동작보다 명시적 동작을 선택한다.
- 최적화는 측정 이후에만 적용한다.

## 2. 네이밍

- 클래스/타입: `PascalCase`
- 함수/변수: `snake_case`
- 상수: `UPPER_SNAKE_CASE`
- 파일명: `snake_case` (예: `create_user_usecase.py`)
- 불리언 변수: `is/has/can/should` 접두어 사용

## 3. 파일/모듈 규칙

- 순환 참조를 금지한다.
- 외부 I/O(DB, HTTP, 파일)는 인프라 계층으로 한정한다.
- 공통 유틸은 `shared`에 두되, 도메인 의미를 가지면 도메인으로 이동한다.
- 한 파일은 하나의 주요 개념(핸들러, 유스케이스, 리포지토리 구현 등)에 집중한다.

## 4. API 규약

- 엔드포인트는 복수형 리소스 명사를 사용한다. 예: `/users`, `/orders/{id}`
- 상태 코드는 의미에 맞게 사용한다. 예: `200`, `201`, `204`, `400`, `401`, `403`, `404`, `409`, `422`, `500`
- 실패 응답은 표준 에러 스키마(`code`, `message`, `requestId`)를 따른다.
- 하위 호환성을 깨는 변경은 버전으로 분리한다. 예: `/v1`, `/v2`
- FastAPI `response_model`을 명시해 응답 스키마를 고정한다.
- 요청 스키마는 Pydantic 모델에서 `extra="forbid"`를 기본으로 검토한다.

## 5. 에러/예외 처리

- 예상 가능한 비즈니스 오류는 명시적 도메인 에러로 처리한다.
- 예외를 무시하지 않는다. 최소 `warn` 로그로 기록한다.
- 외부 호출 실패는 타임아웃/재시도/서킷브레이커 정책을 명시한다.

## 6. 로깅

- 로그는 구조화(JSON) 형태를 기본으로 한다.
- `error` 로그에는 오류 코드, requestId, 호출 경계 정보를 포함한다.
- 개인정보/민감정보는 마스킹하거나 로깅하지 않는다.

## 7. 테스트 규칙

- 단위 테스트: 도메인 규칙과 유스케이스 중심
- 통합 테스트: DB/외부 연동 경계 검증
- E2E 테스트: 핵심 사용자 시나리오 우선
- 버그 수정 시 회귀 테스트를 함께 추가한다.
- 테스트 도구 기본값: `pytest`, `pytest-asyncio`, `httpx.AsyncClient`

## 8. Python 품질 도구

- 포맷터: `black`
- import 정렬: `isort`
- 린트: `ruff`
- 타입체크: `mypy` (점진 도입 가능)

## 9. Git/PR 규칙

- 브랜치: `feature/*`, `fix/*`, `chore/*`
- 커밋 메시지(권장): Conventional Commits
- PR에는 아래를 포함한다.
  - 변경 이유
  - 주요 변경점
  - 테스트 결과
  - 호환성 영향(있다면)

## 10. 코드 리뷰 체크리스트

- [ ] 도메인 규칙이 인프라 코드에 섞이지 않았는가
- [ ] 실패/예외/경계 조건이 테스트되었는가
- [ ] API/스키마 변경 시 하위 호환성을 확인했는가
- [ ] 관측성(로그/메트릭/트레이스) 누락이 없는가
- [ ] 보안/권한 검증 누락이 없는가
- [ ] FastAPI 의존성 주입(Depends) 남용으로 결합도가 높아지지 않았는가
